# Дизайн: Веб-интерфейс конвертера Google Keep → ENEX

**Дата:** 2026-06-28
**Цель:** Добавить простой локальный веб-интерфейс к существующему конвертеру `convert.py`. Пользователь открывает страницу в браузере, загружает файлы выгрузки Keep (.zip архив или перетаскиванием файлов), получает готовый `.enex` для импорта в Apple Notes. Без установки зависимостей.

## Контекст

Готовый конвертер `convert.py` (Python 3, stdlib) обрабатывает папку выгрузки Keep → `.enex`. Веб-интерфейс — тонкий слой поверх него: приём файлов через браузер → подготовка временной папки → вызов существующей логики → отдача результата. Дублирования логики конверсии нет.

## Решения (выбрано на этапе брейншторма)

1. **Подача данных:** загрузка через браузер.
2. **Формат загрузки:** и `.zip` архив, и drag&drop файлов (при наличии zip приоритет у zip).
3. **Веб-стек:** Python 3 stdlib (`http.server`), ноль зависимостей.
4. **Запуск:** `python3 server.py [--port 8000] [--host localhost]`, автооткрытие браузера на `http://localhost:8000`.

## Архитектура (Подход A)

Тонкий веб-слой поверх готовой логики. Чистое разделение ответственности:

```
convert.py            ← рефакторинг: extract convert_directory() (ядро, переиспользуемое)
server.py             ← новый: HTTP-сервер (stdlib http.server)
static/index.html     ← новый: UI (HTML+CSS+JS, один файл, vanilla)
```

- `convert.py` остаётся работающим CLI-инструментом + его функции используют тесты.
- `server.py` — тонкий слой: раздаёт статику, принимает файлы, готовит временную папку, вызывает `convert_directory()`, отдаёт `.enex`.
- `static/index.html` — статичная страница, vanilla JS, без сборки.

## HTTP API сервера (`server.py`, stdlib `http.server`)

| Метод | Путь | Назначение |
|---|---|---|
| GET | `/` | Отдаёт `static/index.html` |
| GET | `/static/*` | Раздаёт статику |
| POST | `/convert` | Принимает multipart-данные (файлы ИЛИ zip) → конвертирует → отдаёт `.enex` |

Запуск: `python3 server.py [--port 8000] [--host localhost]`. По умолчанию слушает только `localhost` (не торчит в сеть). При старте автоматически открывает `http://localhost:PORT` в браузере (`webbrowser`).

## Поток обработки `POST /convert`

```
1. Парсинг multipart-формы (поле "files" — один или несколько файлов; поле "include_trashed")
2. Временная папка (tempfile.mkdtemp)
3. Нормализация входа → единая папка Keep:
     - если есть .zip        → распаковать (zipfile.extractall) во временную папку
     - иначе (drag&drop)     → записать все файлы во временную папку под исходными именами
4. find_keep_root(): рекурсивный общий родитель всех найденных *.json
5. convert_directory(keep_dir=root, output_path=<temp.enex>, include_trashed=...)
6. Ответ: 200 + .enex (application/octet-stream, attachment)
          ИЛИ 400/500 + JSON {error, detail}
7. Удаление временной папки (finally) — независимо от успеха/ошибки
```

### Обработка режимов ввода

**Один .zip:** пользователь сжимает папку «Google Keep» в zip (Finder → «Сжать»). Возможны структуры: `Google Keep/*.json`, `Takeout/Google Keep/*.json`, либо `*.json` в корне архива. Сервер распаковывает и через `find_keep_root()` определяет корень — единая эвристика работает для любой глубины вложенности.

**drag&drop файлов:** пользователь выделяет все файлы в Finder (`.json` + `.jpg/.png`) и тащит в окно. Каждый файл пишется во временную папку под **исходным именем** (имена картинок должны совпадать с `attachments[].filePath` в JSON — имя берётся из multipart-формы). Имена санитизируются (`sanitize_filename`) от path-traversal (`..`, абсолютные пути).

**Приоритет zip:** если в загрузке есть `.zip`, фронтенд отправляет только zip'ы, отдельные файлы игнорируются (с пояснением в UI). Проще пользователя и сервера.

### find_keep_root() — эвристика

`glob` по `tempdir/**/*.json` (рекурсивно). Корень Keep = общий родитель всех найденных `.json`. Если `.json` в корне `tempdir` — корень = `tempdir`. Если все в `tempdir/Google Keep/` — корень = `tempdir/Google Keep`. Надёжно для любой структуры zip.

### Multipart-парсинг

Используется `cgi.FieldStorage` (доступен в Python 3.9.6 — целевой версии проекта). Обёрнут в `try/except`: при отсутствии/ошибке `cgi` возвращается понятная ошибка 500. Для локального инструмента на 3.9 это достаточно и не требует собственного multipart-парсера.

## Фронтенд (`static/index.html`)

Единый файл HTML+CSS+JS (vanilla). Три состояния: «ожидание ввода» → «конвертация» → «результат/ошибка».

**Состояние «ожидание»:**
```
Заголовок: "Google Keep → Apple Notes (.enex)"
[ dropzone: «Перетащите файлы сюда или нажмите Выбрать» ]
   подсказка: «.zip архив или .json + картинки»
[Выбрать файлы]
Список выбранных файлов (с ✕ для удаления)
☐ Включить удалённые (корзину)
[ Конвертировать ]  ← disabled, пока нет файлов
```

**Состояние «конвертация»:**
```
⏳ Конвертация... (не закрывайте страницу)
[ кнопка disabled, спиннер ]
```

**Состояние «результат»:**
```
✅ Готово! Обработано 1505 заметок, 313 вложений, 41 МБ
[ Скачать .enex ]   ← автоскачивание + кнопка
[ Конвертировать ещё ]  ← сброс
```

**Состояние «ошибка»:**
```
❌ Ошибка: <{error} (+ detail)>
[ Попробовать снова ]
```

### Логика фронтенда (vanilla JS)

- **Drag&drop:** события `dragover`/`drop` собирают `dataTransfer.files`, объединяются с файлами из `<input type=file multiple>`.
- **ZIP-приоритет:** если в выбранных файлах есть `.zip` — фронтенд отправляет только zip'ы (UI поясняет: «загружен архив, отдельные файлы игнорируются»).
- **Отправка:** `fetch('/convert', {method:'POST', body: FormData})`. FormData: поля `files[]` + `include_trashed`.
- **Распознавание ответа:** по `Content-Type`. `application/octet-stream` → blob → автоскачивание (`GoogleKeep.enex`) + показ сводки; `application/json` → парс `{error, detail}` → показ ошибки.
- **Сводка результата:** из `Content-Length` (размер) — остальные цифры (заметки/вложения) сервер передаёт в кастомных заголовках ответа `X-Notes`, `X-Attachments`.

## Обработка ошибок (end-to-end)

- Сетевая ошибка fetch → «Не удалось связаться с сервером».
- 400/500 → показ `{error}` (+ короткий `detail`).
- Серверная паника → перехват в `do_POST`, 500 с общим текстом, лог в консоль сервера.
- Временная папка всегда удаляется в `finally`.

## Лимиты

- Размер запроса: `MAX_BODY_SIZE = 200 МБ` в `BaseHTTPRequestHandler`. Превышение → 413 `{error: "Превышен лимит размера (200 МБ)"}`.
- Таймаут сокета: 300 с.
- Значения — константы в начале `server.py`.

## Структура `server.py`

1. `sanitize_filename(name)` — защита от path-traversal.
2. `extract_zip_to(zip_path, dest)` — распаковка + возврат пути корня Keep.
3. `collect_files_to_dir(field_storage, dest)` — запись multipart-файлов в папку.
4. `find_keep_root(tempdir)` — поиск папки с `.json` (рекурсивный общий родитель).
5. `handle_convert(handler)` — оркестрация: tempdir → collect/extract → find_root → `convert_directory()` → отдача файла/ошибки.
6. `KeepHandler(BaseHTTPRequestHandler)` — `do_GET` (статика), `do_POST` (`/convert`).
7. `main()` — argparse (`--port`, `--host`), запуск `ThreadingHTTPServer`, `webbrowser.open()`.

## Edge cases

| Случай | Обработка |
|---|---|
| Не загружено ни одного `.json` | 400 `{error: "Не найдено ни одного .json файла"}` |
| Битый/не-zip файл с расширением .zip | 400 `{error: "Не удалось распаковать архив"}` |
| Файл-картинка без парного `.json` | игнорируется (`convert_directory` работает по glob `.json`) |
| `.json` с path-traversal именем (`../x`) | `sanitize_filename`, отклонение опасных имён |
| Конвертация упала (исключение) | 500 `{error, detail}`, временная папка удаляется |
| Объём > 200 МБ | 413 |

## Тестирование (TDD, тот же assert-раннер)

**Unit-тесты на серверные функции:**
- `sanitize_filename()` → отсекает `..`, абсолютные пути; сохраняет обычные имена.
- `find_keep_root()` → находит корень при вложенной (`tempdir/Google Keep/*.json`) и плоской (`tempdir/*.json`) структурах.
- `extract_zip_to()` → распаковывает тестовый zip (создаётся в фикстуре на лету), возвращает правильный корень.

**Интеграционный тест (HTTP):**
- Запускается `ThreadingHTTPServer` на случайном порту в фоновом потоке.
- `GET /` → 200, тело содержит `<html`.
- `POST /convert` с zip (4 фикстуры `tests/fixtures/` запакованы на лету в zip в памяти) → 200, `Content-Type: application/octet-stream`, тело содержит `<en-export>` и `☐ молоко`/`☑ хлеб`, парсится как XML.
- `POST /convert` без файлов → 400 `{error}`.

**Тест фронтенда:** вручную (открыть в браузере). Selenium/Playwright не подключаются (YAGNI).

## Не входит в рамки (YAGNI)

- Реальный прогресс-бар по файлам (передача — один синхронный запрос; показываем спиннер «Конвертация…»).
- Аутентификация/логин (локальный инструмент, слушает localhost).
- Загрузка по частям/ресумабельность.
- Асинхронная фоновая очередь конвертаций.
- Деплой на удалённый сервер (только локальный запуск).
- Отдельная сборка фронтенда (одна статичная HTML-страница).
